# 嵌入式软件工程师（ESP32方向）面试题

> 基于候选人简历项目「基于 ESP32 的智能环境监测系统」定制
> 技术栈：ESP32-S3 + ESP-IDF + FreeRTOS + MQTT + I2C/ADC

---

## 一、FreeRTOS 基础（必答题）

### 1. 任务调度
**Q: 你的项目中有4个任务（sensor_task、upload_task、display_task、power_task），请解释为什么 sensor_task 优先级最高（5），而 power_task 最低（2）？如果 sensor_task 和 upload_task 同时就绪，会发生什么？**

参考答案要点：
- sensor_task 优先级最高因为传感器采集有实时性要求，DHT11 时序窗口微秒级，错过就要重来
- upload_task 中等优先级，网络上传允许一定延迟
- power_task 最低，因为它是后台监控，不紧急
- FreeRTOS 优先级抢占：高优先级任务就绪时会立即抢占低优先级任务
- sensor_task(5) 就绪时会抢占 upload_task(4)，但 sensor_task 执行完或阻塞后 upload_task 才能运行

### 2. 队列通信
**Q: 项目中 sensor_task 通过队列把数据发给 upload_task 和 display_task。如果队列满了怎么办？你代码里是怎么处理的？**

参考答案要点：
- 代码中队列满时先 `xQueueReceive` 丢弃最旧数据，再 `xQueueSend` 写入新数据
- 这是"丢弃旧数据"策略，适合实时传感器场景（旧数据没价值）
- 另外可以用 `xQueueOverwrite` 覆盖写（但只适合单元素队列）
- 还可以设置超时 `xQueueSend(..., pdMS_TO_TICKS(100))` 阻塞等待

### 3. 互斥锁
**Q: 项目中 BH1750 和 SSD1306 共享 I2C 总线，你用互斥锁保护。请解释：为什么不能用二值信号量（Binary Semaphore）代替互斥锁？**

参考答案要点：
- 互斥锁有**优先级继承**机制：低优先级任务持有锁时，如果有高优先级任务在等锁，低优先级任务会被临时提升优先级，避免优先级反转
- 二值信号量没有优先级继承
- 场景：display_task(Pri3) 持有 I2C 锁 → sensor_task(Pri5) 等锁 → 如果没有优先级继承，sensor_task 会被中间优先级的任务（如 upload_task Pri4）抢占，导致 display_task 迟迟无法释放锁 → sensor_task 饿死
- 互斥锁必须由持有者释放，信号量可以由任何任务释放

## 二、ESP32 硬件与驱动

### 4. DHT11 驱动
**Q: 你的 DHT11 驱动使用了 `taskENTER_CRITICAL` 临界区。请解释：为什么读取 DHT11 时需要临界区？如果不在临界区里读会怎样？**

参考答案要点：
- DHT11 是单总线协议，靠电平持续时间区分 0 和 1（50us LOW + 26-28us HIGH = 0, 50us LOW + 70us HIGH = 1）
- 读取时必须精确计时，如果被 FreeRTOS 调度器中断打断，计时就会出错
- 不在临界区：假设读到第 15 位时被切换到另一个任务，回来时电平已经变了，数据全错
- `taskENTER_CRITICAL` 禁用中断，保证位操作的原子性
- 代价：临界区期间其他任务无法调度，所以要尽量缩短临界区时间

### 5. I2C 总线
**Q: ESP32-S3 的 I2C 主机驱动，`i2c_master_bus_add_device` 和 `i2c_master_transmit` 的区别是什么？如果两个设备地址相同会怎样？**

参考答案要点：
- `i2c_master_bus_add_device` 是在总线上注册一个设备，返回设备句柄，只做配置不通信
- `i2c_master_transmit` 通过设备句柄发送数据，实际产生 I2C 时序
- 地址相同会导致总线冲突，两个设备同时响应，数据线电平混乱
- 解决：改硬件（ADDR 引脚接不同电平）或用两个 I2C 总线

### 6. ADC 采集
**Q: MQ135 用 ADC 采集，你用了 `ADC_ATTEN_DB_12` 衰减。这个参数的含义是什么？为什么不用 `ADC_ATTEN_DB_0`？**

参考答案要点：
- `ADC_ATTEN_DB_12` 表示 12dB 衰减，输入电压范围扩展到 0-3.3V（满量程）
- `ADC_ATTEN_DB_0` 无衰减，满量程只有约 0-1.1V
- MQ135 输出电压范围 0-3.3V，用无衰减会超出量程，读数饱和
- 衰减越大，可测范围越宽，但精度（分辨率）会降低
- ESP32-S3 ADC 是 12 位，4096 个量化级别

## 三、系统设计

### 7. 双核调度
**Q: 你把 sensor_task 绑定到 Core 1，upload_task 绑定到 Core 0。为什么这样分配？如果都放在同一个核心会怎样？**

参考答案要点：
- ESP32-S3 是 Xtensa LX7 双核处理器
- Wi-Fi/TCP 协议栈运行在 Core 0，upload_task 需要调用 MQTT（依赖 TCP），绑定 Core 0 减少跨核通信开销
- sensor_task 对实时性要求高（DHT11 时序），绑定 Core 1 避免被网络中断干扰
- 同核心问题：网络中断可能打断传感器读取时序，导致 DHT11 读取失败
- 双核的好处：真正并行执行，sensor 和 upload 互不阻塞

### 8. 低功耗设计
**Q: ESP32 有几种睡眠模式？你选了 Light-sleep 而不是 Deep-sleep，为什么？Light-sleep 唤醒后系统状态如何恢复？**

参考答案要点：
- ESP32 有 3 种睡眠：Active、Light-sleep、Deep-sleep
- Light-sleep：CPU 暂停，外设可配置保持，RTC 内存保留，~0.8mA，唤醒快（毫秒级）
- Deep-sleep：大部分掉电，只有 RTC 域工作，~10μA，唤醒相当于重启（丢失 RAM）
- 选 Light-sleep 因为：需要保持 I2C 外设状态、唤醒后继续采集不需要重新初始化
- 唤醒后：从 `esp_light_sleep_start()` 返回，CPU 从断点继续执行，所有变量保持不变

### 9. 看门狗问题
**Q: 你的简历提到"网络任务频繁阻塞导致 TWDT 复位"。请解释 TWDT 是什么？你是怎么解决的？**

参考答案要点：
- TWDT = Task Watchdog Timer，监控任务是否长时间不喂狗
- 默认超时 5 秒，如果某个任务 5 秒内没调用 `esp_task_wdt_reset()`，系统复位
- 网络重连可能阻塞几十秒（Wi-Fi 扫描 + DHCP + MQTT 连接），触发 TWDT
- 解决方案：
  1. 降低网络任务优先级，让它可以被抢占
  2. 用非阻塞式网络调用 + 事件组同步
  3. 在网络等待循环中定期调用 `vTaskDelay(1)` 让出 CPU（间接喂狗）
  4. 或者将网络任务排除出 TWDT 监控：`esp_task_wdt_delete(xTaskGetCurrentTaskHandle())`

## 四、MQTT 与云端

### 10. MQTT 认证
**Q: 阿里云 IoT 的 MQTT 认证需要 HMAC-SHA256 签名。请画出密码计算的流程。如果时间戳不对会怎样？**

参考答案要点：
```
clientId  = "DeviceName|securemode=2,signmethod=hmacsha256|"
username  = "DeviceName&ProductKey"
content   = "clientId{clientId}deviceName{dn}productKey{pk}ts{ts}"
password  = HMAC-SHA256(content, DeviceSecret) → 64位hex字符串
```
- 时间戳用于防重放攻击
- 如果时间戳偏差太大（通常允许 ±15 分钟），服务器拒绝连接
- 生产环境应该用 NTP 同步时间，当前代码用固定时间戳（面试时可提及是简化实现）

### 11. QoS 等级
**Q: 你的代码中 publish 用了 QoS 1，subscribe 也用了 QoS 1。请解释 MQTT 的 QoS 0/1/2 区别，为什么选 QoS 1？**

参考答案要点：
- QoS 0：最多一次，可能丢失（传感器数据偶尔丢一条没关系，但命令不行）
- QoS 1：至少一次，保证送达但可能重复（传感器数据重复可以覆盖，命令重复也安全）
- QoS 2：恰好一次，最可靠但开销大（4次握手），嵌入式设备资源有限不划算
- 选 QoS 1：平衡可靠性和资源消耗，传感器数据重复不怕，命令有幂等性

## 五、C 语言与数据结构

### 12. 结构体对齐
**Q: 你的 `sensor_data_t` 结构体有 float 和 uint32_t 成员。这个结构体在 ESP32 上占多少字节？如果调整成员顺序，大小会变吗？**

参考答案要点：
```c
typedef struct {
    float temperature;      // 4 bytes, offset 0
    float humidity;         // 4 bytes, offset 4
    float light_intensity;  // 4 bytes, offset 8
    uint16_t air_quality;   // 2 bytes, offset 12
    uint32_t timestamp;     // 4 bytes, offset 16 (需要4字节对齐，padding 2字节)
} sensor_data_t;            // 总计 20 bytes
```
- ESP32 是 32 位处理器，4 字节对齐
- 当前顺序：float(4) + float(4) + float(4) + uint16(2) + padding(2) + uint32(4) = 20
- 如果把 uint16_t 放最后：float×3(12) + uint32(4) + uint16(2) + padding(2) = 20
- 但如果 float×3(12) + uint16(2) + uint16(2) + uint32(4) = 20，没有浪费

### 13. 指针与 volatile
**Q: 你的 power_manager.c 中 `s_idle_counter` 用了 `volatile` 关键字。为什么？如果去掉会怎样？**

参考答案要点：
- `volatile` 告诉编译器：这个变量可能被意外修改，不要优化掉对它的读写
- s_idle_counter 在 power_task 中递增，在 power_manager_reset_idle() 中被其他任务清零
- 如果去掉 volatile：编译器可能优化成只读一次到寄存器，后续循环中不再从内存读取
- 结果：其他任务清零了 s_idle_counter，但 power_task 看到的还是旧值，永远不进入睡眠

## 六、CMake 与构建系统

### 14. 组件注册
**Q: 你的 components 目录下每个组件都有自己的 CMakeLists.txt。请解释 `idf_component_register` 中 SRCS、INCLUDE_DIRS、REQUIRES 分别是什么作用？**

参考答案要点：
- SRCS：该组件的源文件列表，编译时会编译这些 .c 文件
- INCLUDE_DIRS：头文件搜索路径，其他组件 include 这个组件的头文件时用
- REQUIRES：依赖的其他组件，CMake 会自动处理依赖链和链接顺序
- 例如 dht11 组件 REQUIRES driver esp_timer，说明它用到了 GPIO 驱动和 esp_timer 库
- 如果漏写 REQUIRES，编译时会报 undefined reference 或找不到头文件

## 七、场景题

### 15. 故障排查
**Q: 假设你的环境监测系统运行了 24 小时后，OLED 显示突然卡死，但串口日志显示传感器数据还在正常采集和上传。你怎么排查？**

参考答案要点（考察系统思维）：
1. 先看串口日志有没有 display_task 相关的错误（I2C 通信失败？）
2. 可能原因：
   - I2C 总线死锁：互斥锁没释放（某个任务异常退出但没 Give）
   - SSD1306 初始化丢失：I2C 噪声导致 OLED 进入未知状态
   - 栈溢出：display_task 栈空间不够，局部变量太多
3. 排查步骤：
   - 加 `uxTaskGetStackHighWaterMark()` 检查栈使用
   - 加 I2C 通信超时和错误计数
   - 看互斥锁持有时间是否异常
   - 考虑加看门狗监控 display_task 心跳
4. 解决：I2C 通信失败时重新初始化 OLED，加错误恢复机制
